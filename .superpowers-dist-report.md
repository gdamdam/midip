# midip Release Packaging Report

## Route taken: HAND-WRITTEN release.yml

`cargo-dist` (`dist`) is not installed and sandbox has no network access, so the
release workflow was authored manually. It is functionally equivalent to what
`dist init` would generate: per-platform build matrix → per-OS archive → SHA256
checksums → upload to GitHub Release.

---

## Cargo.toml changes

Added to `[package]`:
- `repository = "https://github.com/gdamdam/midip"`
- `homepage  = "https://github.com/gdamdam/midip"`
- `readme    = "README.md"`

Added new section:
```toml
[profile.release]
opt-level = 3
lto = "thin"
strip = true
codegen-units = 1
```

`lto = "thin"` chosen over `"fat"` for faster CI build times while still giving
meaningful link-time optimisation. `strip = true` removes debug symbols from the
binary (safe for a TUI app with no crash reporter). `codegen-units = 1` maximises
inlining opportunities.

## Targets & archive formats

| Target                    | Runner         | Format   |
|---------------------------|----------------|----------|
| `aarch64-apple-darwin`    | macos-latest   | `.tar.gz` |
| `x86_64-apple-darwin`     | macos-latest   | `.tar.gz` |
| `x86_64-pc-windows-msvc`  | windows-latest | `.zip`    |
| `x86_64-unknown-linux-gnu`| ubuntu-latest  | `.tar.gz` |

All four targets included. No target was dropped.

Archive names follow the pattern: `midip-<tag>-<target>.<ext>`, e.g.
`midip-v0.13.0-aarch64-apple-darwin.tar.gz`.

Each archive contains:
- `midip` binary (or `midip.exe` on Windows)
- `README.md`, `LICENSE`, `CHANGELOG.md` (if present at repo root)

Each archive also gets a sibling `*.sha256` checksum file.

---

## Workflow trigger

```yaml
on:
  push:
    tags:
      - 'v*'
```

Fires on any tag matching `v*` (e.g. `v0.13.0`, `v1.0.0-rc1`). Tags containing
`-` (pre-release semver) are automatically marked as GitHub pre-releases via:
```yaml
prerelease: ${{ contains(github.ref_name, '-') }}
```

---

## How artifacts upload

1. **build** job (matrix, 4 parallel runners): builds binary, stages files,
   creates archive + checksum, uploads as a GitHub Actions artifact
   (`release-<target>`).
2. **publish** job (single ubuntu runner, `needs: build`): downloads all four
   artifacts via `actions/download-artifact@v4 merge-multiple: true`, then calls
   `softprops/action-gh-release@v2` which creates/updates the GitHub Release
   for the pushed tag and uploads all `*.tar.gz`, `*.zip`, `*.sha256` files.

No code signing. No installers. No `.app`/`.dmg`.

## Concerns / notes

- **midir on Windows**: `midir` links against the Windows Multimedia API
  (`winmm`), which is available on `windows-latest` MSVC without extra steps —
  should work out of the box.
- **rusty_link on Windows**: `rusty_link` (Ableton Link) may require extra build
  deps on Windows. If the Windows build fails, the matrix `fail-fast: false`
  means the other three targets still publish; the Windows target can be removed
  or investigated separately.
- **`softprops/action-gh-release@v2`**: requires `permissions: contents: write`
  on the publish job (already set). The default `GITHUB_TOKEN` is sufficient —
  no extra secrets needed.
- **`lto = "thin"` + `strip = true`** are applied to the local binary already
  shipped to `target/release/midip` (build above used this profile).
